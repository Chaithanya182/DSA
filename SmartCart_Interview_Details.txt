============================================================
       SMARTCART — MERN PROJECT INTERVIEW DETAILS
============================================================

PROJECT OVERVIEW
----------------
SmartCart is a full-stack e-commerce platform built with the MERN
stack (MongoDB, Express.js, React, Node.js). It is an online clothing
store with a customer-facing frontend, a separate admin panel, and a
REST API backend.

Deployment: Vercel (frontend, admin, and backend all deployed separately)
Repository: https://github.com/Chaithanya182/SmartCart


============================================================

                      Axios, React Toastify

============================================================
DATABASE DESIGN (3 MongoDB Models)
============================================================

1. USER MODEL
   - name        : String (required)
   - email       : String (required, unique)
   - password    : String (required, bcrypt hashed)
   - cartData    : Object (default: {}, minimize: false)

   Design Decision: Cart is embedded inside the User document
   (not a separate collection) because cart is always accessed in
   context of a user and is small in size — avoids extra DB round trip.
   { minimize: false } prevents Mongoose from stripping the empty {}
   cart object.

2. PRODUCT MODEL
   - name        : String (required)
   - description : String (required)
   - price       : Number (required)
   - image       : Array  (Cloudinary URLs, required)
   - category    : String (Men/Women/Kids, required)
   - subCategory : String (Topwear/Bottomwear/Winterwear, required)
   - sizes       : Array  (required)
   - bestseller  : Boolean
   - date        : Number (Date.now() timestamp, required)

3. ORDER MODEL
   - userId        : String (required)
   - items         : Array  (cart items snapshot, required)
   - amount        : Number (required)
   - address       : Object (delivery details, required)
   - status        : String (default: 'Order Placed')
   - paymentMethod : String (COD/Stripe/Razorpay, required)
   - payment       : Boolean (default: false — marks if payment confirmed)
   - date          : Number (Date.now() timestamp, required)

============================================================
AUTHENTICATION FLOW
============================================================

USER AUTH:
  1. User submits email + password
  2. Backend finds user by email in DB
  3. bcrypt.compare(plain password, stored hash)
  4. On match: jwt.sign({ id: user._id }, JWT_SECRET)  [no expiry set]
  5. Token returned to frontend, stored in state/localStorage
  6. Every protected request sends token in request headers
  7. authUser middleware: jwt.verify(token) --> injects userId into req.body

ADMIN AUTH:
  1. Admin submits email + password (hardcoded in .env)
  2. If they match: jwt.sign(email+password, JWT_SECRET)
  3. Token stored in admin app state
  4. adminAuth middleware: verifies token_decode === email+password string

Key difference: User tokens carry a MongoDB _id payload (object).
Admin tokens carry a plain string payload (email+password concatenated).


============================================================
PAYMENT FLOWS
============================================================

COD (Cash on Delivery):
  1. Create order in DB with payment: false
  2. Clear user cart immediately
  3. payment stays false (COD means pay on delivery)

STRIPE:
  1. Create order in DB with payment: false
  2. Build line_items array from cart items (price * 100 for paise/cents)
  3. Add delivery charge as separate line item
  4. Create Stripe Checkout Session --> get session.url
  5. Frontend redirects to Stripe hosted page
  6. On success: Stripe redirects to /verify?success=true&orderId=xxx
  7. verifyStripe: sets payment: true, clears cart

RAZORPAY:
  1. Create order in DB with payment: false
  2. Create Razorpay order with amount * 100
  3. order._id used as receipt for mapping
  4. Frontend opens Razorpay checkout UI
  5. On success: verifyRazorpay fetches order status from Razorpay API
  6. If 'paid': sets payment: true, clears cart


============================================================
IMAGE UPLOAD FLOW
============================================================

1. Admin submits product form with up to 4 images
2. Multer middleware parses multipart/form-data (fields: image1-image4)
3. Files temporarily stored on server disk
4. Promise.all: Upload all images to Cloudinary in parallel
5. Cloudinary returns secure_url for each image
6. URLs stored in product.image array in MongoDB
7. Frontend fetches and renders Cloudinary CDN URLs directly


============================================================
STATE MANAGEMENT (Frontend)
============================================================

Used React Context API (ShopContext) — choice over Redux because:
- Project scale doesn't justify Redux boilerplate
- Context API is sufficient for shared state: products, cart, token

ShopContext holds:
- products list (fetched on app load)
- cartItems (synced with backend)
- token (user session)
- currency, delivery_fee
- Helper functions: addToCart, updateQuantity, getCartCount, getCartAmount

============================================================
KNOWN WEAKNESSES (MENTION PROACTIVELY IN INTERVIEWS)
============================================================

1. JWT tokens have NO expiry set — should add expiresIn + refresh tokens
2. Admin auth uses string comparison (email+password) not a proper role system
3. Cart stored in User document — doesn't scale well for large carts
4. No inventory/stock tracking — concurrent purchases not handled
5. listProducts fetches ALL products — needs pagination for scale
6. No webhook for Stripe (relies on redirect URL — less reliable)
7. No input sanitization beyond basic validation

============================================================
LIKELY INTERVIEW QUESTIONS + MODEL ANSWERS
============================================================

Q: "Walk me through a Stripe payment flow in your project."
A: When the user clicks Pay with Stripe, the frontend calls POST /api/order/stripe.
   The backend creates an order in MongoDB with payment: false, then builds a
   line_items array from the cart (price * 100 for smallest currency unit), adds a
   delivery charge as an extra line item, and creates a Stripe Checkout Session.
   The session URL is returned to the frontend which redirects the user. After
   payment, Stripe redirects to /verify?success=true&orderId=xxx. My verifyStripe
   endpoint updates payment to true and clears the user's cart.

Q: "Why did you use { minimize: false } on the user schema?"
A: Mongoose strips empty nested objects by default during save. Since cartData
   starts as {}, without minimize: false, Mongoose would remove it and future
   cart reads/writes would throw errors because the field wouldn't exist.

Q: "How do you differentiate user vs admin middleware?"
A: Two separate middleware files. authUser verifies the JWT and extracts the
   user's MongoDB _id from the token payload. adminAuth verifies the token and
   checks that the decoded string matches the hardcoded admin email+password
   from environment variables. Route files apply the correct middleware per endpoint.

Q: "What happens if Stripe succeeds but verifyStripe API call fails?"
A: The order remains payment: false and cart is not cleared — the user's UX
   breaks. The proper fix is Stripe Webhooks, where Stripe pushes a server-side
   confirmation, making it reliable even if the browser redirect fails.

Q: "How would you handle 100,000 concurrent users?"
A: Separate cart into its own collection with indexed userId. Add Redis for product
   caching. Paginate listProducts. Use Stripe webhooks. Add horizontal scaling
   with PM2 cluster mode and a load balancer. Add rate limiting.

Q: "How do you prevent a user from seeing another user's orders?"
A: The userId in userOrders is extracted from the verified JWT token by the
   authUser middleware, not from the request body the user sends. So even if
   a user manipulates the request, they can only fetch orders matching their
   authenticated token's userId.
